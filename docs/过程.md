# 需求整理过程

## 1. 背景梳理

当前项目是一个基于 Vue 3、Vite、TypeScript 的轻量 Web 前端，核心页面集中在 `src/views/Live` 目录。

从现有代码可以判断，项目目标是提供一个 HTTP-FLV 直播播放页面，允许用户维护摄像头配置，并通过 `flv.js` 在浏览器中播放直播流。

经需求沟通确认，该页面最终运行在 uni-app 内的 WebView 中，不作为独立后台系统使用。主要使用者是普通用户，系统属于多人使用场景，但摄像头配置仍然保存在本地。

后续需求沟通中确认，页面需要从单一直播配置页扩展为带底部导航的 WebView 应用。底部导航包含“与我分享”、“直播”、“回看”、“我的”四个入口，其中“与我分享”承载当前通过本地配置观看直播的能力。

现有页面没有明显的多模块业务结构，主流程集中在：

- 展示直播列表
- 检查摄像头在线状态
- 播放 / 停止 HTTP-FLV 视频流
- 本地维护摄像头配置
- 导入 / 复制配置

## 2. 已识别的现有能力

### 2.1 播放能力

系统当前支持为每个摄像头配置一个 FLV 视频地址，并通过 `flv.js` 创建播放器后挂载到 HTML5 `video` 元素。

如果配置了媒体信息接口，页面会先请求该接口，获取是否在线、是否有音频等信息；如果没有配置媒体信息接口，则默认认为该直播源可播放且包含音频。

### 2.2 摄像头配置能力

摄像头配置保存在浏览器 `localStorage` 中。

每个摄像头配置包含：

- `id`：本地唯一标识
- `name`：摄像头名称
- `mediaInfoUrl`：媒体信息请求地址
- `url`：视频播放地址

页面提供添加、删除、编辑、恢复默认配置等能力。

### 2.3 配置导入导出能力

当前页面支持：

- 将本地配置复制到剪贴板
- 将 JSON 格式配置粘贴导入

导入后系统会重新生成每条配置的本地 `id`。

### 2.4 部署方式线索

Vite 构建配置将产物输出到 `../rtmp2flv_app/src/static/live`，说明该前端大概率不是独立部署为公网 Web 站点，而是作为另一个应用的静态资源或内嵌页面使用。

## 3. 需求整理方法

本次需求整理先以现有代码能力为基础，不直接引入新实现，也不修改源码。

整理时采用以下顺序：

1. 识别项目当前已经实现的功能。
2. 抽象出当前系统真正服务的用户目标。
3. 将现有能力整理为明确的功能需求。
4. 补充当前代码中暴露出来但尚未明确的边界条件。
5. 将暂不确定的内容列为待确认问题。
6. 将明显不属于本阶段的事项列为暂不纳入范围。

## 4. 关键判断

### 4.1 项目不是完整后台系统

当前项目没有登录页、用户管理、服务端配置管理或权限页面。虽然 Axios 封装里存在 token、401、403 等处理逻辑，但直播页面本身并没有使用完整认证流程。

因此本阶段需求应聚焦在 WebView 内普通用户直播观看、历史查看、本地配置和个人信息展示，不把它扩展为完整管理后台。

### 4.2 配置数据目前属于本地数据

摄像头配置目前保存在浏览器 `localStorage`，没有看到服务端持久化接口。因此需求中先按“本地配置”定义，不默认要求后端保存。

经确认，即使系统属于多人使用场景，摄像头配置仍然采用本地保存方式。也就是说，不同用户、不同设备或不同 WebView 环境中的配置不会自动同步；如需迁移配置，应通过导入 / 复制配置完成。

### 4.3 媒体信息接口是可选能力

现有逻辑允许摄像头不配置媒体信息接口。没有该接口时，页面默认直播源在线，并直接尝试播放。

因此需求中需要保留两种模式：

- 有媒体信息接口：先检查状态再播放
- 无媒体信息接口：直接播放

### 4.4 当前部署环境可能依赖固定目录

构建产物输出目录指向兄弟项目，说明后续如果调整构建配置，需要确认外部项目的集成方式。本阶段需求文档只记录该事实，不提出代码改动。

经确认，页面运行环境是 uni-app WebView，因此后续需求需要关注 WebView 环境中的播放能力、剪贴板能力、HTTPS 限制、本地存储行为和页面尺寸适配。

### 4.5 功能结构从单页扩展为底部导航

新增需求将页面结构调整为底部三栏导航：

- “与我分享”：保留当前通过配置进行直播观看的页面。
- “直播”：展示主播 card，每个 card 提供播放、停止两个操作。
- “回看”：按直播大厅、主播、历史直播文件列表展示回看内容。
- “我的”：展示个人信息，并提供登录、退出功能。

“回看”页面展示主播直播历史文件列表，点击历史文件后通过弹窗播放该历史文件。

## 5. 已确认事项

以下内容已在本轮需求沟通中确认：

1. 页面运行在 uni-app 里的 WebView 中。
2. 主要使用者是普通用户，不是开发者或专职后台管理员。
3. 系统属于多人使用场景。
4. 摄像头配置使用本地保存，不要求服务端统一管理。
5. 本地配置意味着不同用户或设备之间默认不自动同步配置。
6. 底部导航包含“与我分享”、“直播”、“回看”、“我的”四个入口。
7. “与我分享”是当前通过本地配置进行直播观看的页面。
8. “直播”页面展示主播 card，每个 card 包含播放、停止两个按钮。
9. 点击主播 card 的播放按钮，应播放对应主播直播；点击停止按钮，应停止对应直播。
10. “回看”页面按直播大厅、主播、历史直播文件列表展示。
11. 点击历史文件后弹窗播放历史文件。
12. “我的”页面展示个人信息，并包含登录、退出功能。
13. 个人信息至少包含账号、用户名等内容。
14. 需要支持 HTTPS 视频流和自签名证书场景。
15. “与我分享”页面需要支持多个摄像头同时播放。
16. “直播”页面主播 card 数据从后端接口查询，查询时携带“我的”页面登录后的 token。
17. 主播直播播放地址和历史文件列表由后端接口返回。
18. 历史文件播放格式为 HTTP-FLV。
19. 普通用户可以直接编辑“与我分享”的摄像头地址。
20. 页面需要适配手机竖屏、手机横屏和平板尺寸。
21. 登录、退出由当前 WebView 页面直接完成。
22. 登录状态不需要和 uni-app 外层系统共享。
23. 异常场景可以展示更详细的错误原因，例如连接失败、跨域失败、视频格式不支持等。
24. 配置导入暂不做地址格式和字段完整性校验。
25. 需要保留默认摄像头配置。
26. 本地配置在多人场景下不需要按登录用户隔离缓存。
27. 媒体信息接口的标准返回格式固定为 `{ hasAudio, onlineStatus }`。
28. token 保存位置使用当前项目已有的 `sessionStorage` 方案。
29. 自签名证书暂不在页面侧做校验。
30. 登录接口、主播列表接口、主播直播信息接口、主播历史文件列表接口和退出接口的具体路径、请求参数、返回格式先在代码中预留，后续再补充。
31. “回看”页面的直播大厅查询接口为 `/clientInfo/aq`。
32. “回看”页面的主播列表查询接口为 `/clientCamera/aq/:idClient`。
33. “回看”页面的历史文件查询接口为 `/clientCameraRecord/aqPage/:idClient`。
