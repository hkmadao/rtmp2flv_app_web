# HTTP-FLV 直播播放页面需求说明

## 1. 项目目标

提供一个运行在 uni-app WebView 中的轻量直播观看应用。用户可以通过底部导航进入“与我分享”、“直播”、“回看”、“我的”四个页面，完成本地配置直播观看、主播直播观看、历史文件播放和个人信息管理。

系统优先满足普通用户在多人系统中的直播查看需求。“与我分享”中的摄像头配置保存在本地，不在本阶段扩展为服务端统一配置管理或完整后台管理系统。

## 2. 使用对象

主要使用对象为普通用户。

系统属于多人使用场景，但摄像头配置使用本地保存。不同用户、不同设备或不同 WebView 环境中的配置默认互不自动同步。

本地摄像头配置不按登录用户隔离。用户登录或退出后，“与我分享”中的本地配置仍以当前设备和当前 WebView 存储环境为准。

典型使用场景：

- 在局域网或指定服务环境中查看摄像头直播
- 临时添加或修改摄像头播放地址
- 导入已有摄像头配置
- 复制当前配置用于备份或迁移
- 在 uni-app 内部通过 WebView 打开直播页面
- 查看主播列表并播放主播直播
- 查看主播历史直播文件并弹窗播放
- 查看个人信息，并进行登录或退出

## 3. 功能需求

### 3.1 底部导航

系统应提供底部导航栏，包含以下四个导航入口：

- 与我分享
- 直播
- 回看
- 我的

用户点击底部导航项时，应切换到对应页面。

底部导航应适配 uni-app WebView 中的移动端使用场景，并保持当前页面的主要操作区域不被遮挡。

### 3.2 与我分享页面

“与我分享”页面是当前通过配置进行直播观看的页面。

系统应展示已配置的摄像头列表。

每个摄像头应展示：

- 摄像头名称
- 在线状态
- 是否有音频
- 播放按钮
- 停止按钮
- 视频播放区域

“与我分享”页面应支持多个摄像头同时播放。

### 3.3 在线状态检查

当摄像头配置了媒体信息请求地址时，系统应请求该地址获取媒体状态。

媒体状态至少包含：

- `onlineStatus`：是否在线
- `hasAudio`：是否有音频

当摄像头未配置媒体信息请求地址时，系统可以默认该摄像头可播放，并直接尝试播放视频流。

### 3.4 与我分享直播播放

用户点击播放按钮或视频区域时，系统应尝试播放对应摄像头的 HTTP-FLV 视频流。

播放前，如果存在媒体信息请求地址，系统应先确认摄像头在线；如果摄像头不在线，应提示用户无法播放。

播放时，系统应根据媒体信息中的 `hasAudio` 配置播放器。

### 3.5 与我分享直播停止

直播开始播放后，用户应可以主动停止播放。

停止时，系统应释放播放器相关资源，避免持续占用网络、内存或视频解码资源。

### 3.6 摄像头配置管理

“与我分享”页面应提供配置模式，允许用户维护摄像头配置。

每个摄像头配置应包含：

- 名称
- 媒体信息请求地址
- 视频播放地址

用户应可以：

- 新增摄像头配置
- 编辑摄像头名称
- 编辑媒体信息请求地址
- 编辑视频播放地址
- 删除摄像头配置
- 恢复默认配置

普通用户可以直接编辑摄像头地址，不需要将配置能力隐藏到管理员入口中。

### 3.7 配置导入

系统应允许用户粘贴 JSON 配置并导入。

导入配置后，系统应以导入内容替换当前本地配置。

导入时应为每条配置生成新的本地唯一标识，避免和已有配置产生 ID 冲突。

本阶段导入配置暂不做地址格式和字段完整性校验，但应避免非法 JSON 导致页面整体崩溃。

### 3.8 配置导出

系统应允许用户将当前摄像头配置复制到剪贴板。

复制内容应为 JSON 格式，便于用户备份、迁移或再次导入。

### 3.9 本地保存

摄像头配置应保存在浏览器本地存储中。

用户刷新页面后，系统应继续使用上一次保存的摄像头配置。

当本地没有配置时，系统应初始化并保留默认配置。

在多人使用场景下，本地保存意味着配置以当前设备、当前 WebView 存储环境为准，不要求自动同步到其他用户或其他设备。

### 3.10 直播页面

“直播”页面应展示主播列表。

主播列表数据应从后端接口查询。查询时应携带“我的”页面登录后获得的 token。

直播大厅数据应从后端接口 `/clientInfo/aq` 查询，数据结构为 `TClientInfo`。

主播列表应按直播大厅分组展示。

每个主播应以 card 形式展示。主播 card 应包含主播基本信息和以下操作按钮：

- 播放
- 停止

点击播放按钮时，系统应播放该主播对应的直播。

主播直播播放地址由后端接口返回。

点击停止按钮时，系统应停止该主播正在播放的直播，并释放播放器相关资源。

### 3.11 回看页面

“回看”页面应按“直播大厅 → 主播 → 历史直播文件列表”的层级展示。

直播大厅列表查询接口为 `/clientInfo/aq`，数据结构为 `TClientInfo`。

主播列表查询接口为 `/clientCamera/aq/:idClient`。

历史直播文件列表查询接口为 `/clientCameraRecord/aqPage/:idClient`。

历史文件列表至少应支持展示：

- 历史文件名称
- 直播或文件时间
- 可播放状态

用户点击某个历史文件时，系统应以弹窗形式播放该历史文件。

历史文件播放格式为 HTTP-FLV。

弹窗应提供关闭能力。关闭弹窗时，系统应停止历史文件播放并释放播放器相关资源。

### 3.12 我的页面

“我的”页面用于展示当前用户个人信息，并提供登录、退出功能。

登录、退出由当前 WebView 页面直接完成，不依赖 uni-app 外层登录能力。

登录成功后，页面应保存后端返回的 token，并在查询主播列表、主播直播信息和主播历史文件列表时携带该 token。

token 应使用当前项目已有的 `sessionStorage` 方案保存。

登录状态不需要和 uni-app 外层系统共享。

个人信息至少应包含：

- 账号
- 用户名

当用户未登录时，页面应提供登录入口。

当用户已登录时，页面应展示个人信息，并提供退出功能。

退出后，页面应清理当前登录状态，并返回未登录展示状态。

## 4. 非功能需求

### 4.1 易用性

页面应尽量减少操作步骤。

用户进入页面后，应能通过底部导航清楚地进入“与我分享”、“直播”、“回看”和“我的”四个主要功能区。

配置模式和播放模式应有明确切换入口。

由于主要使用者是普通用户，页面文案和异常提示应避免开发术语，优先说明用户可以采取的下一步操作。

主播 card、历史文件列表、登录和退出入口应清晰易点，适合移动端触控操作。

页面应适配手机竖屏、手机横屏和平板尺寸。

### 4.2 稳定性

播放器停止、页面切换或组件销毁时，应释放定时器、播放器实例和媒体资源。

媒体状态检查失败时，页面不应整体崩溃，应保留可继续操作的状态。

用户切换底部导航、关闭历史播放弹窗或退出登录时，不应遗留正在运行的无效播放器实例。

### 4.3 可维护性

环境地址、默认配置和接口地址不应长期硬编码在业务逻辑中。

后续如需多环境部署，应支持通过环境配置区分开发、测试和生产服务地址。

### 4.4 兼容性

页面运行在 uni-app WebView 中，依赖 WebView 对 HTML5 video、Media Source Extensions 和 HTTP-FLV 播放能力的支持。

不支持相关能力的浏览器应给出可理解的提示。

后续实现和测试应覆盖目标 uni-app 运行环境，尤其是 Android WebView、iOS WebView 或实际目标设备中的播放表现。

页面需要支持 HTTPS 视频流和自签名证书场景。自签名证书的信任链处理方式需要结合目标 WebView 和系统环境确认。

自签名证书暂不在页面侧做额外校验。

### 4.5 安全性

导入配置时应避免因非法 JSON 或缺失字段导致页面崩溃。

如果页面运行在 HTTPS 环境，视频流、媒体信息接口和剪贴板能力需要满足浏览器安全限制。

由于配置保存在本地，配置中包含的摄像头地址不应默认上传到服务端。

## 5. 数据结构

摄像头配置数据结构：

```ts
type LiveInfo = {
  id: string;
  name: string;
  mediaInfoUrl?: string;
  url: string;
}
```

主播数据结构：

```ts
type AnchorInfo = {
  id: string;
  name: string;
  liveUrl: string;
  coverUrl?: string;
  onlineStatus?: boolean;
}
```

主播历史文件数据结构：

```ts
type LiveHistoryFile = {
  id: string;
  anchorId: string;
  name: string;
  url: string;
  startTime?: string;
  endTime?: string;
}
```

用户信息数据结构：

```ts
type UserInfo = {
  account: string;
  username: string;
}
```

登录状态数据结构：

```ts
type LoginSession = {
  token: string;
  userInfo: UserInfo;
}
```

媒体信息接口期望返回：

```ts
type FlvMediaInfo = {
  hasAudio: boolean;
  onlineStatus: boolean;
}
```

登录接口、退出接口、主播列表接口、主播直播信息接口和主播历史文件列表接口的具体路径、请求参数、返回格式先在代码中预留，后续再补充。

## 6. 异常与提示需求

系统应至少处理以下异常场景：

- 媒体信息接口请求失败
- 摄像头不在线
- 视频地址为空
- 播放器创建失败
- FLV 播放过程中报错
- 主播列表获取失败
- 主播直播播放失败
- 历史文件列表获取失败
- 历史文件播放失败
- 导入配置 JSON 格式错误
- 复制到剪贴板失败
- 登录失败
- 退出失败
- token 不存在或登录已过期

异常发生时，应向用户展示明确提示，并尽量不影响其他摄像头或其他主播的操作。

错误提示可以展示更详细的失败原因，例如连接失败、跨域失败、视频格式不支持、登录已过期等。

## 7. 部署需求

当前页面预期运行在 uni-app WebView 中。构建产物可能由 uni-app 或其关联应用作为静态资源加载。

后续正式调整部署方式前，需要确认：

- 构建产物是否仍由 uni-app 或关联外部应用承载
- 是否需要相对路径部署
- 是否需要兼容 WebView
- 接口地址是否由外部应用注入

## 8. 暂不纳入本阶段范围

以下内容暂不作为本阶段需求：

- 权限管理
- 服务端摄像头配置管理
- 摄像头分组
- 完整录像检索和复杂回放控制
- 云台控制
- 截图或录制
- 多画面布局管理
- 告警事件展示
- 国际化
